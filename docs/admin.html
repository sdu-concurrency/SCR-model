🎯 Overview

About PocketBase

PocketBase is the backend service for this project. It stores all data in an embedded SQLite database at back/pb_data/ and exposes two interfaces:

  • Admin UI - http://127.0.0.1:8090/_/ - a browser-based dashboard for managing collections, records, and settings
  • REST API - http://127.0.0.1:8090/api/ - used by the frontend

Starting the backend

To start the backend locally, run from the back/ directory: 

go run main.go serve --http=127.0.0.1:8090

First-Time Admin Setup

No admin account is pre-created. On the very first startup, PocketBase detects that no superuser exists and shows a setup wizard at /_/. Complete the wizard to create your own admin account with the email and password of your choice.

Choose a strong, unique password — this account has full access to all data and settings.

Two separate authentication tiers

This project uses two distinct authentication systems that must not be confused:

Account type Collection Purpose Who uses it
PocketBase admin _superusers Full backend access via /_/ Platform administrators only
Session manager app_users with role = super Create and run assessment sessions via the web app Researchers and facilitators

⚠️ Important Note

The web application refers to app_users records with role = super as "super users". This is an application-level concept and is completely separate from the PocketBase _superusers collection.

🗄️ PocketBase Basics

Admin UI

The PocketBase admin UI is accessible at http://127.0.0.1:8090/_/. After logging in you will see the Collections panel on the left and the record table for the selected collection in the main area.

Key collections

Collection Purpose
_superusers PocketBase admin accounts - full backend access
app_users Application user accounts; role is normal (participant) or super (session manager)
questions Question set definitions containing vulnerability and capability JSON schemas
sessions Assessment session records, each referencing one questions record
tokens Short-lived login tokens for session participants
surveys Completed survey responses
survey_schemas Survey form structure

The pwordtext field

app_users records have a helper field called pwordtext. When a session is first created, the backend generates a random 5-character password and stores it in both the real PocketBase password hash and in pwordtext so the session manager can see it on screen.

Once the session manager logs in and sets their own password via the web app (the /auth-set endpoint), the real password hash is updated and pwordtext is cleared to an empty string.

⚠️ Important

pwordtext is only a convenience display field. The real authentication secret is always the PocketBase password hash. If you edit only pwordtext, you are not changing the account password.

🔐 Authentication Tiers Explained

PocketBase Superusers (_superusers)

  • Full backend access via /_/
  • Can manage all collections, records, and settings
  • Used only by platform administrators
  • Email + password authentication
  • No relationship to application session managers

Application Super Users (app_users with role = super)

  • Session managers who create and facilitate assessment sessions
  • Access the web application, not the PocketBase admin UI
  • Token-based authentication (auto-generated on session creation)
  • Can create sessions, view results, manage participants

👤 Create an Admin Account

When to create

Create a new PocketBase admin account when you need to give another person direct access to the backend UI at /_/. Admin accounts live in the _superusers collection and have no relationship with application session managers.

Steps

  1. Open http://127.0.0.1:8090/_/ and log in with an existing admin account.
  2. In the left sidebar, click _superusers under the System section.
  3. Click + New Record in the top-right corner.
  4. Enter the new admin's email address and a strong password.
  5. Click Create to save the record.
  6. Share the credentials with the new admin securely.
  7. Ask the new admin to log in and change their password immediately.

Expected result

The new account can log in to /_/ and manage all collections, records, and settings.

⚠️ Warning

Admin accounts have unrestricted write access to the database. Only create accounts for people who need this level of access.

Alternative: create via CLI

You can also create a superuser directly from the command line without opening the UI - useful on first deployment or when the UI is inaccessible.

Local / development (from the back/ directory):

go run main.go superuser create EMAIL PASS

Docker (replace <container> with your container name or ID):

docker exec <container> /pb/pocketbase superuser create EMAIL PASS

See the PocketBase production docs for more details.

🔄 Reset a Super User Password

Overview

In this project a "super user" is an app_users record with role = super. These accounts are session managers who create and facilitate assessment sessions through the web application.

The application assigns a random password when a session is created. If a session manager forgets their password, you can reset it from the PocketBase admin UI by editing the pwordtext field of the record - this triggers the reset password mechanism on the next login.

Understanding the two password fields

When you open an app_users record in PocketBase you will see:

  • change password toggle - reveals the Password and Confirm fields when enabled
  • pwordtext - the plaintext helper field (only shown to admins)

To fully reset the password you must update both:

  1. The real PocketBase password (via the change password toggle)
  2. The pwordtext field to match, so the session manager can enter the system until they set their own

Steps

  1. Open http://127.0.0.1:8090/_/ and log in.
  2. Click app_users in the left sidebar.
  3. Find the session manager's record - look for role = super in the role column. You can also filter by typing role = "super" in the search bar.
  4. Click the record row to open the edit panel.
  5. Scroll down and verify that role is set to super.

  6. Option A: Toggle change password to enable the password fields.

    1. Enter the new password in both Password and Confirm.
    2. Click Save changes.

  7. Option B: Update the pwordtext field to the new password value.

    1. This approach will trigger reset password mechanism on the next login.
    2. Click Save changes.
  8. Provide the new password to the session manager via a secure channel.
  9. Ask them to log in and set their own permanent password through the web application.

➕ Add a New Vulnerability and Capability Set

Overview

Assessment question sets are stored in the questions collection. Each record defines a complete survey configuration including a vulnerability schema and a capability schema. The session creation form in the web application loads all records from this collection and presents them as options in the Session question set dropdown.

Record fields

Field Required Validation Notes
name Yes Any text Identifies the question set, e.g. supply_chain or cybersecurity
version Yes Must match v1, v2, v3, etc. Incremented when revising an existing set
description No Any text Short human-readable summary shown in the UI
vulnerability_schema Yes Valid JSON array Categories and items for the vulnerability assessment
capability_schema Yes Valid JSON array Categories and items for the capability assessment

The combination of name and version should be unique across all records.

JSON schema structure

Both schemas follow the same structure: a JSON array of category objects. Each category has a multilingual label and an items array. Each item has a stable value key (used as the internal identifier in survey responses) and a multilingual label. 

[
  {
    "label": { "en": "1. Finance", "da": "1. \u00d8konomi/finans" },
    "items": [
      {
        "value": "S1_1",
        "label": { "en": "Asset turnover", "da": "Aktivers oms\u00e6tningshastighed" }
      },
      {
        "value": "S1_2",
        "label": { "en": "Access to liquidity", "da": "Adgang til likviditet" }
      }
    ]
  },
  {
    "label": { "en": "2. Customers / demand", "da": "2. Kunder/eftersp\u00f8rgslen" },
    "items": [
      {
        "value": "S2_1",
        "label": { "en": "Concentration of customers", "da": "Kundekoncentration" }
      }
    ]
  }
]

⚠️ Important

The value keys are persisted in survey response records. Do not change existing value keys after surveys have been collected - doing so will break the mapping between responses and questions.

Steps to add a new question set

  1. Open http://127.0.0.1:8090/_/ and log in.
  2. Click questions in the left sidebar.
  3. Click + New Record in the top-right corner.
  4. Fill in the name field (e.g. cybersecurity).
  5. Fill in the version field using the format v1.
  6. Optionally fill in description with a short note about the question set.
  7. Click the vulnerability_schema field and replace the null value with your JSON array.
  8. Click the capability_schema field and replace the null value with your JSON array.
  9. Click Create to save the record.
  10. After saving, navigate to the web application at http://localhost:5173/admin, log in, and open the session creation form. The new question set should appear immediately in the Session question set dropdown.

📊 Note

The frontend loads question sets sorted by creation date with the newest first. The most recently added question set will appear at the top of the dropdown and will be pre-selected by default.

✅ Verification Checklist

After making changes in PocketBase, work through this checklist before handing the system back:

New admin account

  • ✅ Log out of the current admin session
  • ✅ Log back in using the new admin account credentials
  • ✅ Confirm that collections and records are accessible

Super user password reset

  • ✅ Confirm that pwordtext in the app_users record matches the new password
  • ✅ Test logging in to the web application using the session manager's username and the new password
  • ✅ The session manager should be prompted to set their own permanent password on first login

New question set

  • ✅ The questions record is visible in the collection list view
  • ✅ The name and version are correct
  • ✅ Both vulnerability_schema and capability_schema contain valid JSON arrays
  • ✅ The new question set name appears in the session creation dropdown at http://localhost:5173/admin
  • ✅ Create a test session using the new question set and confirm it is created successfully

❓ Troubleshooting

New question set does not appear in the web UI

  1. Confirm the record was saved - check the questions collection list view.
  2. Reload the session creation page; the frontend queries the API on every page load.
  3. Check browser developer tools for API errors on the /api/collections/questions/records request.

Password reset did not work - session manager cannot log in

Open the app_users record and confirm that change password was toggled on before saving. If only pwordtext was updated, the real password hash was not changed. Toggle change password, re-enter the password, and save again.

JSON editor shows an error

Common causes:

  • Trailing comma after the last item in an array or object
  • Missing quotes around string keys or values
  • null left in the field instead of replaced with a JSON array

The PocketBase JSON editor highlights syntax errors inline. Paste your JSON into a validator (e.g. jsonlint.com) before pasting into PocketBase if you are unsure.

The questions record saves but the frontend shows an error

Check that both schemas are JSON arrays (starting with [), not objects (starting with {). The frontend expects an array of category objects.

📄 JSON Schema Reference

Below are the complete vulnerability_schema and capability_schema values from the default supply_chain / v1 question set. Use these as a reference or starting template when creating a new question set.

Default Vulnerability Schema

The vulnerability schema includes 7 categories with multiple items each, covering all aspects of supply chain vulnerability:

  • Finance: Asset turnover, Access to liquidity, Net working capital, Low cash flow
  • Customers / demand: Customer concentration, Dependence on key customers, Demand volatility
  • Processes / organization: Dependency on key personnel, Limited production flexibility, Single-source production
  • Systems / data: Outdated IT systems, Lack of data visibility, Cybersecurity vulnerabilities
  • Purchasing / sourcing: Supplier concentration, Long lead times, Single-source components
  • Supply chain end-to-end: Geographic concentration, Limited visibility, Complexity
  • Environment: Climate-related risks, Regulatory and compliance risks, Geopolitical risks

Default Capability Schema

The capability schema includes 7 categories with multiple items each, covering organizational capabilities:

  • Finance: Financial reserves, Access to credit facilities, Financial risk management
  • Customers / demand: Customer diversification, Demand forecasting, Customer relationship management
  • Product / processes: Product standardization, Process flexibility, Lean/agile operations, Cross-training
  • Inventory management: Safety stock levels, Inventory visibility, Buffer strategy
  • Sourcing / purchasing: Multi-sourcing strategy, Supplier qualification, Long-term relationships, Local/nearshore sourcing
  • Systems and data: Real-time visibility, ERP/planning system maturity, Data analytics, Cybersecurity posture
  • Management and organization: Executive commitment, Business continuity planning, Crisis response, Supply chain risk culture

The complete JSON schemas with all categories and items are available in the HANDBOOK_BACKEND.md file in the repository.
